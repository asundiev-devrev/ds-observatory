# DS Observatory

Design system observability for DevRev — a CLI that measures **how much of our product is actually built on the design system**, tracks **DLS → Arcade migration progress**, and flags **where designers have drifted away from library components**.

Produces a local dashboard and a single self-contained HTML report.

---

## What it measures

Three headline metrics, computed from live Figma data:

| Metric | What it answers |
|---|---|
| **DS Coverage** | % of component instances across our top design files that come from *any* DS library (vs. one-off primitives / raw frames). |
| **Arcade Adoption** | Of those DS instances, what % are Arcade vs. legacy DLS. This is the migration progress bar. |
| **Detachment Rate** | % of surface that looks like a detached DS component (e.g. a frame named `Button` with no component link). The leak indicator. |

It also ranks **files by Arcade ratio** (who's migrated, who's lagging), lists the **most-used components** in each library, and trends the three headline metrics over time using snapshot history.

---

## How it works

1. **Team catalog.** Paginates `/v1/teams/{id}/components` for the DS team to build a `component_key → (library, display name)` index (~5k published components).
2. **Hot-file discovery.** Finds the N product files most recently modified (default 15, last 60 days) across the product team.
3. **File traversal.** For each hot file, walks the node tree, classifies every `INSTANCE` node against the catalog, and heuristically flags detached frames (a node named like a known component but not linked to one).
4. **Synthesis.** Aggregates per-file breakdowns into library analytics. (Figma's official Library Analytics API is Enterprise-only; we replicate it from traversal instead.)
5. **Snapshotting.** Every run is timestamped in `data/snapshots/` so the dashboard can draw trend lines.

Two consumers of that data:
- `ds-observatory serve` — local dashboard (HTML + JS + Canvas charts, no build step).
- `ds-observatory report` — one self-contained HTML file with all data inlined, deployable to GitHub Pages.

## Prerequisites

- Node.js ≥ 18
- A Figma **personal access token** with read access to the product and DS teams
- IDs for: the product team (hot files come from here), the DS team (component catalog), and the two library files (DLS + Arcade)

## Usage

```bash
# Fetch fresh Figma data (10–20 min — Figma API is slow on large files)
npm run dev -- collect

# Smaller scope if you're debugging or hit timeouts
npm run dev -- collect --hot-files 8 --window 30

# Local dashboard
npm run dev -- serve
# → http://localhost:3333

# Self-contained HTML report for sharing / GitHub Pages
npm run dev -- report -o ds-observatory-report.html
```

## Commands

### `collect`
Fetches Figma data and writes to `./data/`.

| Flag | Default | Description |
|---|---|---|
| `--token <t>` | `$FIGMA_ACCESS_TOKEN` | Figma PAT |
| `--team <id>` | `$FIGMA_TEAM_ID` | Product team (for hot-file discovery) |
| `--hot-files <n>` | `15` | Files to scan |
| `--window <days>` | `60` | Lookback window for "active" files |

Outputs:
- `data/library-analytics.json` — library-level rollup
- `data/hot-file-audit.json` — per-file breakdown (can be multi-MB)
- `data/snapshots/*.json` — timestamped history for trend charts

### `serve`
Starts a static dashboard on `localhost:3333`. Reads straight from `./data/`.

| Flag | Default |
|---|---|
| `-p, --port <port>` | `3333` |

### `report`
Writes a single-file HTML report with all data, fonts, and JS inlined.

| Flag | Default |
|---|---|
| `-o, --output <path>` | `ds-observatory-report.html` |

## Data contract

```
data/
├── canonical-components.json        # static: seed list of DS components
├── library-analytics.json           # generated by `collect`
├── hot-file-audit.json              # generated by `collect`
└── snapshots/
    ├── library-analytics-<iso>.json
    └── hot-file-audit-<iso>.json
```

Run `collect` on a schedule (cron, CI, GitHub Action) to build a trend history. Each run adds one snapshot.

## Known quirks

- **Figma API is the bottleneck.** Large files (>60 pages) are fetched per-page; expect 1–3 min per such file. Transient `TypeError: terminated` errors are auto-retried 3× with exponential backoff, then the file is skipped rather than aborting the whole run.
- **Library Analytics API returns 403** on non-Enterprise plans. The collector falls back to synthesizing the same data from hot-file traversal.
- **Dashboard is zero-build.** Plain HTML/CSS/JS served from `src/dashboard/`. Edit files and refresh.
- **Detachment detection is heuristic.** A frame named `Button` sitting in the middle of a file will be counted as a detachment whether or not it was ever a DS instance.
